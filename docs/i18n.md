## 1. Architecture: two translation systems

ArgoUML loads UI strings from Java `.properties` bundles. Two independent
mechanisms coexist, and they do not share state.

### 1.1 ArgoUML's `Translator.localize(key)`

[`org.argouml.i18n.Translator`](../src/argouml-app/src/org/argouml/i18n/Translator.java)

```java
// Pseudocode (see Translator.java:259 for the actual implementation)
public static String localize(String key) {
    String bundle = key.substring(0, key.indexOf('.'));   // "menu.popup.x" → "menu"
    return ResourceBundle.getBundle("org.argouml.i18n." + bundle, locale)
                         .getString(key);
}
```

- Used by **all ArgoUML code** that displays UI strings.
- Bundle base name is derived from the leading dotted prefix:
  `action.foo` → `action`, `menu.popup.x` → `menu`, `label.x.y` → `label`, …
- If the key is missing in the active locale, lookup walks the fallback chain
  (`zh_CN` → `zh` → default locale → key string itself).

### 1.2 GEF's `Localizer.localize(name, key)`

`org.tigris.gef.util.Localizer` (in `gef-0.13.7.jar`)

```java
public static String localize(String name, String key) {
    Map<String, ResourceBundle> resources = defaultResources.get(defaultLocale);
    ResourceBundle bundle = resources.get(name);  // alias-keyed Map lookup
    if (bundle == null) return key;               // silent fallback to literal
    return bundle.getString(key);
}
```

- Used **inside GEF** (e.g. `Fig.getPopUpActions()` builds the "Ordering"
  popup submenu with `Localizer.localize("PresentationGef", "Ordering")`).
- ArgoUML registers GEF bundles via
  `Translator.java:140-145`:

  ```java
  Localizer.addResource("GefBase",
                        "org.tigris.gef.base.BaseResourceBundle");
  Localizer.addResource("GefPres",
                        "org.tigris.gef.presentation.PresentationResourceBundle");
  ```

- **Name mismatch bug**: ArgoUML registers GEF under `"GefPres"`,
  but GEF code calls `Localizer.localize("PresentationGef", …)`. Lookup
  fails, GEF returns the literal `"Ordering"` regardless of locale.
- **Items inside the submenu** (`Forward / Backward / To Front / To Back`)
  come from `CmdReorder.wordFor()` which returns **hardcoded English literals
  in the GEF bytecode** (no `Localizer` call). Even fixing the name alias
  does not localize these.

**Take-away**: if you see English text in any popup that wasn't put there
by ArgoUML code, it's GEF. Fixing it requires either overriding GEF
behavior or shipping zh_CN bundles inside a patched GEF jar. See
[§6 Limitations](#6-known-limitations).

## 2. Anti-pattern catalog (real code)

Each entry below is a real bug found in the ArgoUML codebase. Pattern,
diagnosis, and fix are shown.

### 2.1 `super(EnglishLiteral)` in `UndoableAction` subclasses

**Symptom**: menu item shows English text regardless of locale.

**Diagnosis**:

```bash
grep -rnE 'super\("[A-Z][a-zA-Z ]+"\)' src/argouml-app/src/org/argouml/ui/foundation
```

**Example**: `ActionSetAssociationEndOrdering.java:64`

```java
// Before — broken in zh_CN: "Set" displayed (en bundle has "Set" but…
// Translator.localize("Set") falls through to literal because "Set" →
// bundle base "Set" → org.argouml.i18n.Set_zh_CN.properties not found)
protected ActionSetAssociationEndOrdering() {
    super(Translator.localize("Set"), null);          // WRONG: "Set" is not in any bundle
    putValue(Action.SHORT_DESCRIPTION,
            Translator.localize("Set"));              // ditto
}

// After — uses an existing key
protected ActionSetAssociationEndOrdering() {
    super(Translator.localize("action.set"), null);   // FIXED
    putValue(Action.SHORT_DESCRIPTION,
            Translator.localize("action.set"));       // FIXED
}
```

**Lesson**: the **bundle base name** is the leading dotted prefix. A bare
word like `"Set"` looks up `org.argouml.i18n.Set` which does not exist —
the key itself is returned as the fallback string. Always use the
`bundle.<key>` form.

### 2.2 `// TODO: I18N` comments around literal strings

**Symptom**: code clearly intended to be localized but never was.

**Diagnosis**:

```bash
grep -rn "// TODO: I18N\|TODO.*i18n\|TODO.*localize" src/
```

**Example**: `FigAssociation.java:792`

```java
// Before — FigOrdering.getOrderingName() returns literal "{ordered}"
private String getOrderingName(Object orderingKind) {
    ...
    if ("ordered".equals(name)) {
        return "{ordered}";   // TODO: I18N
    }
    ...
}

// After
private String getOrderingName(Object orderingKind) {
    ...
    if ("ordered".equals(name)) {
        return "{" + Translator.localize("label.ordered") + "}";
    }
    ...
}
```

**Lesson**: a `// TODO: I18N` comment is a *confirmed bug*, not a wishlist.
Treat as a bug report and fix it.

### 2.3 Duplicate translation values

**Symptom**: two distinct UI strings rendered identically in zh_CN.

**Diagnosis**: compare keys whose English values differ but zh_CN values are
identical:

```bash
diff <(grep -E "^[a-z]" en/action.properties | sort -u) \
     <(grep -E "^[a-z]" zh/action_zh_CN.properties | sort -u)
```

Look for lines where the key's left-hand side differs but the right-hand side
is the same byte sequence. This is almost always a copy-paste mistake.

**Example**: `action_zh_CN.properties:172`

```properties
action.bring-to-front=\u79fb\u81f3\u9876\u5c42    # 移至顶层  ✓
action.send-to-back =\u79fb\u81f3\u9876\u5c42    # 移至顶层  ✗ duplicate
```

The two operations are opposites (`bring-to-front` = move to highest z-layer;
`send-to-back` = move to lowest). Both rendered "移至顶层" until the
duplicate was fixed to `\u79fb\u81f3\u5e95\u5c42` (移至底层).

**Lesson**: a missing translation falls back to the parent key value, but
a *wrong* translation silently passes review. Always cross-check zh_CN
values against intent, not just existence.

### 2.4 zh_CN bundle missing keys that en bundle has

**Symptom**: zh_CN user sees the **en string** for some UI elements.

**Diagnosis**: count keys per bundle:

```bash
echo "en:    $(grep -cE '^[a-z]' en/X.properties)"
echo "zh_CN: $(grep -cE '^[a-z]' zh/X_zh_CN.properties)"
```

When `zh_CN` < `en`, the missing keys are silently falling back. Pick the
`bundle.class` for which the keys are most commonly seen and fill them in.

**Example**: `checkbox_zh_CN.properties` lacks `checkbox.abstract-uc`,
`checkbox.final-uc`, `checkbox.root-uc`, `checkbox.active-uc`. Users see
the literal "Abstract / Leaf / Root / Active" in the Modifiers submenu.

**Lesson**: zh_CN bundles should add **every** key, not just the ones
visible in the screens you've tested. Each Modifiers submenu has 4 entries;
each diagram type has multiple right-click entries — multiply.

### 2.5 GEF hardcoded strings in popup submenus

**Symptom**: a popup submenu (e.g. Ordering, inside most figures' right-click
menu) shows English even when the rest of the UI is Chinese.

**Diagnosis**: trace `popup getPopupActions` source. The English submenu
comes from somewhere outside ArgoUML sources.

```bash
unzip -p /tmp/argouml-deps/gef-0.13.7.jar org/tigris/gef/presentation/Fig.class \
  | javap -c | grep -i "ordering\|reorder\|popup"
```

In `Fig.getPopUpActions()` bytecode you see `Localizer.localize("PresentationGef", "Ordering")`
plus four `add(CmdReorder.BringForward / SendBackward / BringToFront / SendToBack)`.

**Lesson**: GEF-level strings cannot be fixed by editing ArgoUML bundles
alone. See [§6 Limitations](#6-known-limitations) for the full list.

## 3. Pitfalls (Java + MDR + GEF)

### 3.1 Java 9+ Unicode escape preprocessor

**Symptom**: source file won't compile; error mentions "illegal Unicode
escape" with a confusing column number that doesn't match any line in the
file.

**Diagnosis**: the Java compiler's tokenizer reads the source as a stream
**before** it parses tokens. Unicode escapes (`\uXXXX`) are substituted
as one of the first steps. If you write `\u` followed by 4 non-hex chars
(or fewer chars), the preprocessor tries to substitute and fails.

```java
//  WRONG — \u5c42 is 4 hex digits, but the preprocessor sees \u0000
//  and converts it to a NUL byte, breaking the rest of the file.
String s = "底层" + "\u5c42";   // 底层 is a real Chinese string, that's fine.
                               // The issue is when you write literal \u in source,
                               // like a Javadoc {@code \uXXXX} comment.
```

In **Javadoc**, the same rule applies. Don't write `{@code \u53f6}` in a
comment — the compiler will try to substitute it. Use plain text or HTML
entities: `{@code &#x53f6;}` or `{@code 叶}`.

**Lesson**: the byte-trick in tests (§4.1) sidesteps this entirely.

### 3.2 MDR `getName()` requires a real UML element

**Symptom**: test method calls `Model.getFacade().getName(stub)` and gets
`IllegalArgumentException: Must have an MDR element supplied` or
`ClassCastException` from inside MDR.

**Diagnosis**: the MDR backend (`argouml-core-model-mdr`) implements
`Facade.getName()` by casting the argument to `MElement` (the NetBeans MDR
interface). Strings, `Object` stubs, or `null` cause the cast to fail.

**Fix**: pass a real element:

```java
Object orderingKind = Model.getOrderingKind().getOrdered();
String name = Model.getFacade().getName(orderingKind);   // works

// WRONG — String is not an MElement
String name = Model.getFacade().getName("ordered");      // ClassCastException
```

The EUML backend is more permissive but tests should not depend on
backend specifics.

### 3.3 `Class.forName()` for static nested Fig classes

**Symptom**: reflection `Class.forName("Foo$Bar")` succeeds for inner
non-static classes but fails for static nested classes; or vice versa.

**Diagnosis**: in ArgoUML, some Fig classes are **static nested classes**
declared inside another Fig (e.g. `FigOrdering` inside `FigAssociation`).
The compiler generates `FigOrdering.class` (no `$`), not `FigAssociation$FigOrdering.class`,
because the enclosing class has no `this$0` field. The bytecode confirms:

```bash
unzip -p argouml-app.jar org/argouml/uml/diagram/ui/FigOrdering.class | javap -p
# → shows class org.argouml.uml.diagram.ui.FigOrdering (no enclosing class)
```

**Fix**: when reflecting, use the simple name without `$`:

```java
// Works for static nested class FigOrdering
Class<?> c = Class.forName("org.argouml.uml.diagram.ui.FigOrdering");
Constructor<?> ctor = c.getDeclaredConstructor(Object.class, DiagramSettings.class);
ctor.setAccessible(true);
FigOrdering figOrdering = (FigOrdering) ctor.newInstance(owner, settings);
```

### 3.4 `DiagramSettings` must not be null

**Symptom**: `NullPointerException` inside `FigText`/`ArgoFigText`
constructor.

**Diagnosis**: `ArgoFigText.<init>` calls `this.settings.getFontName()`.
If `settings` is null, NPE.

**Fix**:

```java
DiagramSettings settings = ProjectManager.getManager()
    .getCurrentProject()
    .getProjectSettings()
    .getDefaultDiagramSettings();
FigOrdering figOrdering = new FigOrdering(owner, settings);
```

### 3.5 `tearDown()` must release the project

**Symptom**: subsequent tests fail with `ConcurrentModificationException`
inside `ProjectManager`.

**Diagnosis**: each test's `setUp()` calls `Project.makeEmptyProject()`
which registers a project in `ProjectManager`. Without `tearDown` clearing,
projects accumulate and `ProjectManager` walks the list during shutdown.

**Fix**:

```java
protected void tearDown() throws Exception {
    ProjectManager pm = ProjectManager.getManager();
    Project p = pm.getCurrentProject();
    if (p != null) {
        pm.removeProject(p);
    }
    super.tearDown();
}
```

## 4. Test patterns

### 4.1 The `byte[]` trick (avoid Java 9+ Unicode escape preprocessor)

When asserting that `Translator.localize(key, zh_CN)` returns a specific
Chinese string, do **not** write the Chinese character literally in the test
source:

```java
//  WRONG — the literal characters compile fine but are
//  inconsistent across editors and OS file encoding round-trips
assertEquals("\u67e5\u627e", actual);    // OK in this case, but
                                         // \u53f6 directly in source
                                         // can become a preprocessor trap
                                         // when the editor saves raw bytes
```

Use raw `byte[]` ISO-8859-1 bytes:

```java
byte[] expectedBytes = new byte[] {
    0x5c, 0x75, 0x35, 0x33, 0x66, 0x36    // \u53f6 = 叶
};
String expected = new String(expectedBytes, "ISO-8859-1").trim();
String actual = Translator.localize("checkbox.final-uc",
                                    new Locale("zh", "CN"))
                          .replaceAll("\r", "").trim();
assertEquals(expected, actual);
```

The bytes `5c 75 35 33 66 36` are the ASCII encoding of the six characters
`\u53f6`. When ISO-8859-1 decoded, they yield the escape sequence as a
literal string. Then `Translator.localize()` resolves the escape to the
Chinese character `\u53f6` = `叶`. Asserted against the JVM-decoded result.

This pattern works because:
- The source file is ISO-8859-1 (matches the bundle files).
- The test contains no `\u` escape sequences (preprocessor can't trip).
- The byte sequence round-trips through any editor that doesn't mangle
  ASCII.

See `TestAssociationOrderingI18n` for the full pattern with 9 assertions.

### 4.2 Project setUp / tearDown

Tests that touch the UML model need a project context:

```java
public class FooI18nTest extends TestCase {
    private Project project;

    @Override
    protected void setUp() throws Exception {
        super.setUp();
        InitializeModel.initializeDefault();
        project = ProjectManager.getManager().makeEmptyProject();
    }

    @Override
    protected void tearDown() throws Exception {
        ProjectManager pm = ProjectManager.getManager();
        if (project != null) {
            pm.removeProject(project);
        }
        super.tearDown();
    }

    public void testFoo() { ... }
}
```

### 4.3 Loading bundles for assertion

```java
public void testFanyiKeyExists() {
    ResourceBundle bundle = ResourceBundle.getBundle(
        "org.argouml.i18n.checkbox", new Locale("zh", "CN"));
    assertNotNull("checkbox_zh_CN.properties not found", bundle);

    // Assert a key with the byte[] trick
    byte[] expectedBytes = new byte[] { 0x5c, 0x75, 0x36, 0x32, 0x36, 0x31 };  // \u6261
    assertEquals(new String(expectedBytes, "ISO-8859-1").trim(),
                 bundle.getString("checkbox.abstract-uc"));
}
```

### 4.4 Counting bundle fall-throughs

```java
public void testEnBundleHasNoHardcodedFallback() {
    ResourceBundle en = ResourceBundle.getBundle(
        "org.argouml.i18n.action", Locale.ENGLISH);
    Enumeration<String> keys = en.getKeys();
    while (keys.hasMoreElements()) {
        String key = keys.nextElement();
        String value = en.getString(key);
        // No ASCII-uppercase-only fallback like "BROWSE_CRITICS"
        assertFalse("Likely a hardcoded constant, not a key: " + key,
                    key.matches("^[A-Z_]+$"));
    }
}
```

### 4.5 Testing figures and popups via reflection

```java
public void testFigOrderingText() throws Exception {
    DiagramSettings settings = ProjectManager.getManager()
        .getCurrentProject().getProjectSettings().getDefaultDiagramSettings();
    Class<?> cls = Class.forName("org.argouml.uml.diagram.ui.FigOrdering");
    Constructor<?> ctor = cls.getDeclaredConstructor(Object.class, DiagramSettings.class);
    ctor.setAccessible(true);
    Fig fig = (Fig) ctor.newInstance(getOwner(), settings);
    assertTrue(fig instanceof ArgoFigText);   // verify it bound correctly
}
```

## 5. Diagnostic commands

### 5.1 Find hardcoded literals in source

```bash
# String-literal super() calls
grep -rnE 'super\("[A-Z][a-zA-Z .]+"' src/argouml-app/src/

# Hardcoded menu / button text in Swing constructors
grep -rnE 'new ArgoJMenu\("[A-Z]' src/argouml-app/src/

# TODO: I18N comments
grep -rn '// TODO:.*I18N\|// TODO:.*i18n\|// TODO:.*localiz' src/

# super("Set") and similar 1-word lowercase calls
grep -rnE 'super\("[A-Z][a-z]+\"\)' src/argouml-app/src/

# String literal as second arg to putValue
grep -rnE 'putValue\(Action\.[A-Z_]+,\s*"[A-Z]' src/argouml-app/src/
```

### 5.2 Compare bundle key counts

```bash
for b in action label menu checkbox button combobox critics dialog mnemonic optionpane radiogroup statusmsg tab tooltip misc; do
  en=$(grep -cE '^[a-z]' src/argouml-app/src/org/argouml/i18n/${b}.properties 2>/dev/null || echo 0)
  zh=$(grep -cE '^[a-z]' externals/argouml-i18n-zh/src/org/argouml/i18n/${b}_zh_CN.properties 2>/dev/null || echo 0)
  echo "$b  en=$en  zh=$zh"
done
```

A non-zero gap (`zh < en`) means zh_CN falls back to English for those
keys.

### 5.3 Find specific duplicate values within zh_CN

```bash
# Find lines whose VALUES are identical but keys differ
sort externals/argouml-i18n-zh/src/org/argouml/i18n/action_zh_CN.properties \
  | awk -F'=' '
      NR==1 { header=$0; next }
      /^[^#]/ { vals[$2] = (vals[$2] ? vals[$2] "|" $1 : $1) }
      END { for (v in vals) if (split(vals[v], a, "|") > 1) print vals[v], "=>", v }'
```

Catches the kind of copy-paste mistake that escaped the
`action.send-to-back` zh_CN fix.

### 5.4 Find files that reference a key

```bash
grep -rn "Translator.localize(\"action.foo\")" src/argouml-app/src/
```

### 5.5 Verify the running JVM is using the latest bundle

```bash
# After rebuilding the i18n jar:
mvn -f externals/argouml-i18n-zh/pom.xml install -DskipTests -o
cp externals/argouml-i18n-zh/target/argouml-i18n-zh-*.jar /tmp/argouml-deps/
# Restart ArgoUML (the Translator reads bundles at startup, no hot-reload)

nohup bash src/argouml-app/src/bin/run-argouml.sh > /tmp/argouml.log 2>&1 &
```

## 6. Known limitations

These cannot be fixed by editing ArgoUML source alone.

### 6.1 GEF `Fig.getPopUpActions()` "Ordering" submenu

GEF's `Fig.getPopUpActions()` (in `gef-0.13.7.jar`) builds a `JMenu` named
"Ordering" with 4 items:

- Bring Forward
- Send Backward
- Bring To Front
- Send To Back

The `JMenu` name comes from
`Localizer.localize("PresentationGef", "Ordering")`. ArgoUML registers GEF
under the alias `"GefPres"` (Translator.java:145), but GEF code uses
`"PresentationGef"`. Result: name lookup fails, the literal string
`"Ordering"` is shown in every locale.

The 4 items are `CmdReorder.SendBackward / BringForward / BringToFront /
SendToBack` whose display names come from `CmdReorder.wordFor(int)` —
a method that returns hardcoded English literals:

```java
protected static String wordFor(int function) {
    if (function == CmdReorder.SEND_BACKWARD) return "Backward";
    if (function == CmdReorder.SEND_TO_BACK)   return "ToBack";
    if (function == CmdReorder.BRING_FORWARD)  return "Forward";
    if (function == CmdReorder.BRING_TO_FRONT) return "ToFront";
    throw new IllegalArgumentException();
}
```

**No localization mechanism is invoked** at popup-build time. Fixing this
requires one of:

1. **Override `getPopUpActions()` in `FigNodeModelElement` and
   `FigEdgeModelElement`** to filter out GEF's Ordering JMenu and insert a
   localized one (using `ReorderAction(localizedString, icon, constant)`).
   ~30 lines of code. Repeats with the menu bar Arrange → Reorder submenu,
   which already works correctly.
2. **Ship zh_CN bundles for GEF** (`PresentationResourceBundle_zh_CN.properties`
   for the submenu name + `BaseResourceBundle_zh_CN.properties` for the
   items, even though they wouldn't be picked up by `CmdReorder.wordFor()`).
   Even with these in place, the items stay English.
3. **Patch GEF upstream** (dormant since 2010). Not viable.

The menu bar **Arrange → Reorder submenu** does work correctly — it uses
ArgoUML's `ReorderAction(Translator.localize("action.X"), …)` pattern in
`GenericArgoMenuBar.initReorderMenu()`. Only the **per-element right-click
popup** is affected.

Decision recorded in AGENTS.md §Limitations not yet present; will be
addressed in a future batch if user requests.

### 6.2 ShortcutMgr literal action names

`ShortcutMgr.java:731,734,738,741` creates 4 `ReorderAction` with hardcoded
English names:

```java
new ReorderAction("Forward",   ReorderAction.BRING_FORWARD)
new ReorderAction("Backward",  ReorderAction.SEND_BACKWARD)
new ReorderAction("ToFront",   ReorderAction.BRING_TO_FRONT)
new ReorderAction("ToBack",    ReorderAction.SEND_TO_BACK)
```

These are bound to **keyboard accelerators** (Cmd-F, Cmd-B, Cmd-Shift-F,
Cmd-Shift-B), not added to any visible menu. The display string is used
for the mnemonic tooltip and the `Action.NAME` key. Not visible in normal
right-click popups, but `getKeymap().getAction(name)` returns these in
the action map (visible to screen readers and some plugin code).

Fixing requires replacing literals with `Translator.localize("action.X")`.
Lower priority than visible-UI fixes.

### 6.3 `CmdReorder` action singletons

GEF creates `CmdReorder.SendBackward / BringForward / BringToFront /
SendToBack` in its static initializer. Their display names are set from
`wordFor(int)` and never updated. Even if GEF bundles had zh_CN entries,
these singletons already cached the English names.

## 7. Bundle encoding migration (1996 → 2025)

ArgoUML's `.properties` bundles historically used the ISO-8859-1 +
`\uXXXX` convention mandated by Java before JDK 9. As of 2025 (commit
`utf8-migration`), production code loads bundles as **UTF-8** via a
custom `ResourceBundle.Control` subclass.

### What changed

| Aspect | Before | After |
|---|---|---|
| Source file encoding | ISO-8859-1 | UTF-8 |
| Native characters | `\uXXXX` only | Direct UTF-8 OR `\uXXXX` |
| Translator loader | `getBundle(name, locale)` | `getBundle(name, locale, UTF8_CONTROL)` |
| `CheckResourceBundle` | reads with default Control | `UTF8_CONTROL` |
| POM `<project.build.sourceEncoding>` | `ISO-8859-1` | `UTF-8` |
| Author workflow | `native2ascii` required | None |

### What did NOT change

- **No `.properties` file was rewritten.** All 60 zh_CN/zh_TW files
  keep their `\uXXXX` escapes verbatim. Java's `Properties.load()`
  decodes them regardless of source encoding, so both old
  (ISO-8859-1-encoded with `\uXXXX`) and new (UTF-8 with native chars)
  bundles load identically.
- **No translator public API change.** `Translator.localize(key)`
  behaves exactly as before. Existing call sites need no edits.
- **No GEF bundle modifications.** `gef-0.13.7.jar` uses its own
  `Localizer` which is independent of our UTF8_CONTROL.
- **No argouml-ai bundle modifications.** Those files (`ai.properties`,
  `http.properties`) are pure ASCII; they work with any encoding.

### How Translator.UTF8_CONTROL works

```java
private static final ResourceBundle.Control UTF8_CONTROL =
    new ResourceBundle.Control() {
        @Override
        public ResourceBundle newBundle(String baseName, Locale locale,
                                        String format, ClassLoader loader,
                                        boolean reload)
                throws java.io.IOException, IllegalAccessException,
                       InstantiationException {
            if (!"java.properties".equals(format)) {
                return super.newBundle(baseName, locale, format, loader, reload);
            }
            // Open resource stream, read with UTF-8 InputStreamReader,
            // return PropertyResourceBundle populated from the parsed Properties.
            ...
        }
    };
```

JDK 9+ does not expose `ResourceBundle.Control.getControl(Charset)` —
only `getControl(List<String>)`. UTF-8 support requires subclassing
`Control` and overriding `newBundle()` to use a UTF-8
`InputStreamReader`. This pattern is the documented workaround; it is
used by Spring Framework, Eclipse, and other Java i18n libraries.

### Verifying the migration

After restarting ArgoUML:

```bash
# zh_CN API endpoint should return Chinese characters
curl http://localhost:18766/health
# {"ok":true,"data":{"ok":true,"enabled":true,"project":"未命名"}}
#                                                     ^^^^^^^^ "Untitled"

curl http://localhost:18766/project/diagrams
# [{"name":"用例图","kind":"usecase",...},{"name":"类图","kind":"class",...}]
#          ^^^^^^                  ^^^
#     "Use Case Diagram"      "Class Diagram"
```

### Author guidelines (post-migration)

- **New bundle keys**: write native UTF-8 characters directly.
  `# This is a comment` is fine; don't translate it.
- **Modifying existing bundles**: leave `\uXXXX` escapes in place.
  Both formats work; mixing them is fine.
- **New language modules** (`externals/argouml-i18n-<lang>/`): copy
  `i18n-template/`, then write bundles in UTF-8. Use native
  characters for non-Latin scripts.

### Failure modes

If the migration regresses:

| Symptom | Likely cause |
|---|---|
| `MalformedInputException` on bundle load | A `.properties` file is mixed-encoding; check `file <path>` reports UTF-8 |
| `?????` placeholders in zh_CN output | Bundle missing the key, falling back to en |
| All ASCII instead of Chinese | `user.language` system property is wrong; or `init("en")` ran before test |
| Test assertions show wrong language | Translator was not initialized to the test's locale — call `Translator.init(...)` first |

### When to run `native2ascii -reverse` (optional)

The vendored zh bundles are already in **native UTF-8 form**
(as of late 2025). New contributors may write native characters
directly. Legacy `\uXXXX` files are still accepted at runtime but
should be migrated for readability using:

```bash
# scripts/decode-unicode-escapes.py handles 60 files in ~30s
# Preserves CRLF, validates key counts, idempotent
scripts/decode-unicode-escapes.py FILE [FILE ...]
```

For one-off conversion, the script logic:

```python
ESCAPE_RE = re.compile(r"\\u([0-9a-fA-F]{4})")
text = open(path, 'rb').read().decode('iso-8859-1')
decoded = ESCAPE_RE.sub(lambda m: chr(int(m.group(1), 16)), text)
open(path, 'wb').write(decoded.encode('utf-8'))
```

## 8. GEF "Ordering" submenu (zh_CN / zh_TW)

When you right-click a diagram element, GEF 0.13.7 adds an
**"Ordering"** submenu with 4 items: Bring Forward, Send Backward,
Bring To Front, Send To Back. Pre-2025 both the title *and* the 4
inner items were hardcoded English regardless of the JVM locale,
because the names come from string constants baked into GEF's
`CmdReorder` bytecode.

### What was fixed (late 2025)

The fix is implemented in ArgoUML rather than in GEF. Two
mechanisms collaborate:

1. **`FigNodeModelElement.getPopUpActions`** and
   **`FigEdgeModelElement.getPopUpActions`** now call
   `ReorderMenuBuilder.replaceGefOrderingSubmenu`
   after their `super.getPopUpActions(me)` call. The helper:
   - Inspects each `JMenu` returned by GEF and removes it when it has
     exactly 4 children whose underlying action is
     `org.tigris.gef.base.CmdReorder` (the signature of GEF's
     "Ordering" submenu — we match on the shape, not the title, so a
     future GEF with a translated title is still stripped).
   - Appends a freshly built `JMenu` whose four
     `ReorderAction(items, icon, int)` entries are labeled with
     `Translator.localize("action.bring-forward")` and friends.

2. **`GenericArgoMenuBar.initReorderMenu`** (the menu-bar
   *Arrange &gt; Reorder* submenu) now defers item creation to
   `ReorderMenuBuilder.populate`. The build is shared with the popup
   helper so the menu-bar and the per-element right-click always
   agree on labels, accelerators, and icons.

### Resulting locale behaviour

| Locale | Submenu title | 4 items |
|---|---|---|
| `en`   | `Reorder`       | `Bring Forward` / `Send Backward` / `Bring To Front` / `Send To Back` |
| `zh_CN` | `重新排序`     | `上移一层` / `下移一层` / `移至顶层` / `移至底层` |
| `zh_TW` | `重新排序`     | `上移一層` / `下移一層` / `移至最上層` / `移至最下層` |

### Rationale: how GEF resists localisation here

GEF 0.13.7's `CmdReorder.wordFor(int)` returns the 4 inner item
names by reaching into bytecode string constants:

```java
// Decompiled from gef-0.13.7's CmdReorder.class:
protected static String wordFor(int function) {
    if (function == CmdReorder.SEND_BACKWARD) return "Backward";
    if (function == CmdReorder.SEND_TO_BACK)   return "ToBack";
    if (function == CmdReorder.BRING_FORWARD)  return "Forward";
    if (function == CmdReorder.BRING_TO_FRONT) return "ToFront";
    throw new IllegalArgumentException();
}
```

These constants never go through `Localizer.localize`, so even adding
`BaseResourceBundle_zh_*.properties` to the GEF classpath would not
help. The only way to fix the per-item names without forking GEF is
to *not use* GEF's `CmdReorder` singletons in the popup — which is
exactly what the ArgoUML override does (it builds `ReorderAction`s
with `Translator.localize`d names).

### Coverage

`FigNodeModelElement` covers 22 ArgoUML subclasses; `FigEdgeModelElement`
covers 10. Together they cover all ArgoUML-provided diagram element
renderings (classes, use cases, actors, states, sequence-diagram
lifelines, deployment nodes, etc.). External plugin modules that
extend `FigNode`/`FigEdge` directly (without going through the two
bases) bypass our override and would still show the English
"Ordering" — but no such module currently lives in this repo.

### Test coverage

`TestReorderPopupI18n` builds synthetic GEF-shaped and localized
submenus directly via `ReorderMenuBuilder`, verifying that:

- the helper produces a 4-item `JMenu` whose actions are all
  `ReorderAction` (no `CmdReorder` leakage);
- the per-item labels equal `Translator.localize(action.<key>)` in
  the active locale;
- the strip helper removes a GEF-shaped `JMenu` and appends the
  localized variant in its place;
- non-GEF `JMenu`s are preserved.

The full test suite (1214 baseline + 4 new) runs to a green build
on JDK 17 with surefire's `-Djava.awt.headless=true`.
